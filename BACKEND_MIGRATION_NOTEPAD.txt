URU Research / URU Smart Backend Migration Note
================================================

ไฟล์นี้ใช้ส่งให้ฝั่ง Backend/Server เพื่อ migrate ฐานข้อมูลให้ frontend ใช้งานได้ครบ


ล่าสุดที่ต้อง migrate เพิ่ม
---------------------------

ต้องมี migration นี้บน server:

backend-src/database/migrations/2026_07_15_000005_add_owner_user_id_to_researchers_table.php

สิ่งที่ migration นี้ทำ:
- เพิ่มคอลัมน์ researchers.owner_user_id
- ผูก foreign key ไปที่ users.id
- backfill owner_user_id ให้ researchers เดิม โดยเทียบ researchers.email = users.email

เหตุผล:
- เพื่อให้ระบบรู้ว่า "นักวิจัย" record ไหนถูกเพิ่มโดย user คนไหน
- user ทั่วไปจะแก้ไข/ลบได้เฉพาะข้อมูลที่ owner_user_id เป็นของตัวเอง
- admin แก้ไข/ลบได้ทุกข้อมูล
- ถ้า owner_user_id เป็น NULL user ทั่วไปจะแก้ไข/ลบไม่ได้

คำสั่งที่ต้องรันหลัง deploy โค้ดล่าสุด:

php artisan migrate --force
php artisan db:seed --force
php artisan optimize:clear

คำสั่งเช็กหลัง migrate:

SELECT id, name, email, owner_user_id FROM researchers;
SELECT id, title, owner_user_id FROM projects;
SELECT id, title, owner_user_id FROM articles;
SELECT id, title, owner_user_id FROM proposals;
SELECT id, title, owner_user_id FROM reports;

3) หลักการ owner / permission
------------------------------

ทุก record ที่ user สร้าง ต้องบันทึก:

owner_user_id = auth()->id()

ห้ามใช้ชื่อ researcher/author เป็นตัวตัดสิน owner อย่างเดียว
เพราะชื่อซ้ำ/เปลี่ยนชื่อ/พิมพ์ชื่อแทนกันได้

กติกาการลบ:
- admin แก้ไข/ลบข้อมูลได้ทุก record
- user ทั่วไปแก้ไข/ลบได้เฉพาะ record ที่ owner_user_id ตรงกับ user id ของตัวเอง
- ถ้า owner_user_id เป็น NULL user ทั่วไปจะแก้ไข/ลบไม่ได้ ต้องให้ admin จัดการหรือ backfill owner_user_id ก่อน

5) สำคัญสำหรับ IIS Server
--------------------------

ถ้า IIS ไม่รับ PUT / PATCH / DELETE ตรง ๆ
ให้ backend รองรับ method override:

POST /proposals/1
X-HTTP-Method-Override: DELETE

Body:
{
  "_method": "DELETE"
}

Frontend ใส่ method override ให้แล้ว
แต่ backend ต้องอ่าน header/body นี้ด้วย


6) การตรวจหลัง migrate
------------------------

SELECT COUNT(*) FROM users;

SELECT id, name, owner_user_id FROM researchers;

จะทำให้ "ของฉัน" ไม่ขึ้น หรือแก้/ลบไม่ได้

8) ถ้าข้อมูลเก่ามีอยู่แล้ว
---------------------------

ต้อง backfill owner_user_id:

researchers.owner_user_id จาก researchers.email = users.email
projects.owner_user_id จาก projects.researcher = users.name
articles.owner_user_id จาก articles.author = users.name
proposals.owner_user_id จาก proposals.researcher = users.name
reports.owner_user_id จาก reports.researcher = users.name

ถ้าชื่อไม่ตรง ต้อง map เองด้วย user id ที่ถูกต้อง


9) อาการถ้ายังไม่ได้ migrate/seed
-----------------------------------

- หน้า list ไม่มีข้อมูล
- เพิ่มแล้วไม่เห็นรายการ
- กดลบไม่ได้
- หน้า "ของฉัน" ว่าง
- budget แสดงผิด
- upload รูป/ไฟล์ไม่ได้
- API error ว่า column/table ไม่มี
